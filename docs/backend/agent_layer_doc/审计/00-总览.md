# agent_layer 审计总览

## 审计范围

- 设计基线：
  - `docs/backend/architecture.md`
  - `docs/frontend/重构说明.md`
- 代码范围：
  - `backend/app/agent_layer`
  - `backend/app/service_layer/api/assistant_routes.py`
  - `backend/app/service_layer/api/conversation_routes.py`
  - `backend/app/service_layer/bootstrap/container.py`

## 已验证事实

1. `POST /api/v1/query` 是主链入口，`/api/v1/conversations/chat` 现在只做同一条 Runner 的转发。
2. `AppContainer` 现在一次性持有 `ConversationWindowStore`、`EditorContextStore`、`SessionPersistence`，不再每次请求临时 new 会话态。
3. `written_context / selection` 已写入进程内内存，并在 `_freeze_snapshot` 阶段一并进入 snapshot。
4. `compact.py` 已接入 `TurnRunner`，摘要会写回 `SessionPersistence`，并进入下一轮 `_build_messages`。
5. `cache_mode` 统一输出为 `memory`，SSE metadata 不再输出旧缓存字段。
6. `assistant_routes` 不再返回 `degraded`，编辑器上下文接口直接操作内存单例。
7. `conversation_window` 现在是共享内存窗口，不再走旧缓存适配层。
8. `container.turn_runner` 已缓存，避免跨请求重复装配。
9. 长对话 compact 的效果已经可被测试覆盖，且摘要会参与后续上下文组装。

## 总体结论

- 当前 `agent_layer` 已经收口到“内存会话态 + SQLite 长期事实”的方向。
- 现在最值得盯的不是旧缓存兼容，而是：
  - compact 是否持续生效
  - assistant context 是否稳定进入 query
  - SSE / health 文档是否继续和代码保持一致

## 优先级建议

1. 先保回归：
   - `/api/v1/query`
   - `/api/v1/conversations/chat`
   - `assistant context`
2. 再补：
   - compact 长对话
   - retrieval loop
   - reflection
3. 最后继续观测：
   - request 级模型能力
   - 模型列表接口
   - SSE / health 文档一致性
