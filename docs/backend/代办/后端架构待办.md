# 后端架构待办

这份文件不再记录“大而化之的以后再说”，而是记录两类东西：

1. 已经锁定的默认落地方式
2. 仍然需要在实现阶段验证或收尾的少量问题

## 0. 当前不做的事

在三层主链没有重建完成前，当前明确不做：

- 不先接评测体系
- 不先做 RAG A/B
- 不继续把 Kimi 当活动默认值
- 不继续给旧 `api/clients/core/models/pipelines/stores/utils` 补新功能

这些不是永久放弃，而是明确后置。

## 1. 文档导入与解析链路

### 默认落地

- 文档导入采用“探针路由 + 多策略处理链”，不把单一解析器写死为主基线
- `PDF` 与 `DOCX` 在数据契约层面同级支持
- `PDF` 当前默认方向是轻量优先：
  - 轻量探针
  - 轻量文本主链
  - 必要时走图片 / OCR / 公式 / 表格增强 API
- 每次导入统一保存：
  - `markdown`
  - `structured json`
  - `extraction report`
  - 锚点与必要原始证据
- 图片语义增强默认是 best-effort：
  - 有图片就尝试增强
  - 失败进入 `degraded`
  - 不阻断主文本链

相关模块文档：

- [文档导入与预处理模块](../文档导入与预处理模块/README.md)
- [PDF 清洗重构方案](../关于PDF的清洗模块/PDF清洗重构方案.md)

### 仍待验证

- `DOCX` 的模块级实现文档还需要补齐
- PDF 探针字段的阈值还需要真实样本校准
- 远程增强 API 的超时、成本和降级门槛还需要落表

## 2. parent-child chunk 持久化

### 默认落地

- 父块保存正文语义与原文锚点
- 子块保存图片语义描述，并通过 `parent_anchor_id` 回指父块
- 向量化时：
  - 父块一定入索引
  - 子块是否单独入向量索引，取决于其是否具备独立检索价值
- 回答组装时：
  - 父块先入证据包
  - 子块按需联动注入
  - 同一图片语义不能重复注入多次

### 仍待验证

- 哪些图片语义值得单独建向量，而不是只做联动附属块

## 3. 语义切分与 tokenizer

### 默认落地

- 默认先走语义切分
- 标题层级参与 chunk 边界
- 表格、公式、脚注不切断
- `PDF` 与 `DOCX` 共用同一套切分原则
- retrieval chunk 与 answer context pack 分离

tokenizer 默认策略：

- embedding 侧使用与 embedding 模型同名 tokenizer
- 远程 embedding API 也必须本地保留对应 tokenizer，用于预算与校验
- rerank 如果模型不同，则维护独立 tokenizer 预算
- 生成模型 token 统计优先信 provider usage，其次才是本地估算

超长处理顺序：

1. 先重切
2. 再强制均分
3. 仍超限则拒绝入索引

overlap 默认策略：

- 只在单个语义块超过 embedding 模型上下文 `1/4` 时启用
- 首次默认 overlap 取块长的 `10%`
- 若样本验证显示冗余过高，再往下调，不往上无节制加

### 仍待验证

- 你的真实论文样本在 `10% overlap` 下的召回收益是否显著
- retrieval chunk 到 answer context pack 的具体拼装规则是否还要再分层

## 4. reflection 驱动的取证预算

### 默认落地

- 不使用固定 `topk`
- 每轮都要产出 reflection verdict：
  - `supported`
  - `insufficient`
  - `conflicting`
  - `off_track`
- 默认最多 3 轮 reflection
- 默认最多 2 次方向切换
- 命中 `supported` 就停止继续取证
- 连续两轮 `off_track` 直接停止，返回结构化失败或请求用户补充

### 仍待验证

- 哪类问题更适合第一轮就偏 dense，哪类问题第一轮应偏 sparse

## 5. 输入加权与冻结副本

### 默认落地

活动输入固定为：

- `written_context`
- `selection`
- `prompt`

四种起手权重保持当前总纲约定：

- only prompt: `100 / 0 / 0`
- only written_context: `0 / 0 / 100`
- written_context + selection: `0 / 70 / 30`
- written_context + selection + prompt: `50 / 30 / 20`

这里的数字顺序统一写成：

- `prompt / selection / written_context`

冻结副本默认策略：

- 每次用户发送时，以 `request_id` 冻结一份输入快照
- 快照进入当轮 planning / retrieval / generation
- 常驻轮询继续更新 live `written_context`
- 冻结副本默认短 TTL 保存，便于日志排障和失败恢复

### 仍待验证

- 冻结副本是只进 Redis，还是同时落一份 SQLite 调试痕迹

## 6. 上下文 token 监控与 compact

### 默认落地

- 每轮响应开始时计算：
  - `context_tokens`
  - `remaining_tokens`
  - `remaining_ratio`
- 三个值进入首个 SSE metadata
- `remaining_ratio < 5%` 时触发 compact
- compact 使用成本更低的小模型
- compact 摘要写入长期事实层
- 下一轮 prompt 组装顺序默认是：
  - `summary`
  - 最近窗口消息
  - 当前冻结副本

### 仍待验证

- compact 摘要是单段文本，还是结构化摘要块更适合你的使用习惯

## 7. Redis 内存驱动 TTL 与恢复策略

### 默认落地

监控源：

- Redis `INFO memory`
- 主判据 `used_memory / maxmemory`

TTL 策略：

- `<= 90%`：1 小时
- `90% - 95%`：指数级衰减
- `>= 95%`：拒绝新增长期窗口写入

默认公式：

`ttl = max(120s, floor(3600 * exp(-40 * (usage_ratio - 0.90))))`

恢复策略：

- Redis 异常先自动重连
- 重连失败进入 `degraded`
- 只要 Redis 恢复，后续请求重新使用活动窗口

### 仍待验证

- 是否要把 Redis 当前内存占用也暴露成单独健康指标卡片

## 8. 状态机错误语义

### 默认落地

导入链和对话链都必须支持：

- `retrying`
- `degraded`
- `needs_user_action`
- `failed`

默认重试原则：

- 网络类、限流类、临时 I/O 错误：最多 3 次自动重试
- 文档损坏、路径失效、权限错误：直接 `needs_user_action`
- 图片语义失败、rerank 失败、Redis 临时失败：优先 `degraded`

前端错误载荷默认字段：

- `code`
- `message`
- `stage`
- `retryable`
- `degraded`
- `suggested_action`

### 仍待验证

- 具体到每个外部依赖的错误分类表还需要实现时补齐
