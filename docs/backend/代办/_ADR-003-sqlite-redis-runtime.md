# ADR-003: SQLite + Redis 作为正式运行时双存储

## Status

Accepted

## Context

当前项目既需要长期事实，也需要高频临时态：

- 长期事实：文档元数据、导入记录、对话记录、摘要记录
- 临时态：活动窗口、`written_context`、冻结副本、短期缓存

把这些全塞进 SQLite，会让活动窗口读写成本和语义都不对；只靠进程内缓存，又会在重启后丢态。

同时，用户明确说明 Redis 已安装，正常情况下应视为可用，但仍然需要故障降级与内存保护。

## Decision

- SQLite 保存长期事实
- Redis 保存活动窗口和短期运行态
- Redis TTL 不是死值，而是受内存占用驱动
- `90%` 开始缩 TTL，`95%` 以上停止新增长期窗口写入
- Redis 异常优先自动恢复，恢复失败再进入 `degraded`

## Consequences

正面：

- 长期事实与短期态边界清楚
- 常驻轮询、冻结副本、compact 前后工作态有正式归属
- 能把内存 `_poll_cache` 这类进程内伪状态移出活动主链

代价：

- 需要补齐 Redis 监控、重连、退化和日志
- 数据一致性需要按“长期事实优先，短期态可重建”思路设计
